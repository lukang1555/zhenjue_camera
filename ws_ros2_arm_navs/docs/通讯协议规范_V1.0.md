# WebSocket 通讯协议规范

- 文档名称：高空机械臂视觉自动导航 WebSocket 通讯协议规范
- 协议版本：V1.0
- 生效日期：2026-04-12
- 适用范围：PAD、视觉算法、运控算法、机械臂控制系统、系统服务

## 1. 目标与原则

本规范用于统一系统间实时通信格式，确保以下目标：

1. 支持统一消息类型：`payload`、`ping`、`pong`、`ack`。
2. 支持统一外层参数，所有接口复用同一信封结构。
3. `ack` 为通讯层“收到即回”语义，不承载业务处理结果。
4. `code` 为通讯层状态码，非业务状态码。
5. 心跳策略：客户端每 1 秒发送 `ping`，5 秒超时。

## 2. 传输层规范

1. 传输协议：WebSocket（全双工长连接）。
2. 编码格式：UTF-8 JSON。
3. 传输链路：`wss://`（生产环境建议启用 TLS）。
4. 报文大小：单帧不超过 256KB；超大业务数据使用拆分或外部资源引用。

## 3. 统一消息结构（Envelope）

### 3.1 字段顺序

外层字段顺序统一为：

1. `v`
2. `type`
3. `api`
4. `msg_id`
5. `identity`
6. `code`
7. `body`
8. `ext`
9. `ts`

### 3.2 字段定义

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| v | string | 是 | 协议版本，当前固定 `1.0` |
| type | string | 是 | 消息类型：`payload`/`ping`/`pong`/`ack` |
| api | string | 是 | 接口标识；心跳使用 `SYS_HEARTBEAT` |
| msg_id | string | 是 | 全局唯一消息 ID，用于追踪与去重 |
| identity | string | 是 | 发送方身份标识，统一放设备 ID 或令牌标识 |
| code | integer | 是 | 通讯层状态码；业务结果禁止放此字段 |
| body | object | 是 | 业务数据载荷；心跳/ack 可为空对象 `{}` |
| ext | object | 否 | 扩展字段，默认 `{}` |
| ts | integer | 是 | 发送时间戳（Unix 毫秒） |

说明：
- 字段 `identity`，暂定兼容“设备 ID / 令牌标识”两类场景。
- 若后续需拆分身份与认证，可在未来版本增加 `subject_id` 和认证字段（可放 `ext`）。

## 4. 消息类型定义

## 4.1 payload

- 含义：业务数据传输消息。
- 规则：接收方收到后必须立即返回 `ack`。
- 注意：`ack` 仅表示“通讯层已接收”，不表示业务执行成功。

## 4.2 ping

- 含义：心跳探活请求。
- 规则：仅客户端主动发送，服务端收到后立即返回 `pong`。

## 4.3 pong

- 含义：心跳响应。
- 规则：由服务端响应客户端 `ping`。
- 规则：在 `ext.echo_msg_id` 中携带对应 `ping.msg_id`。

## 4.4 ack

- 含义：通讯层确认消息。
- 语义：收到后回复（收到即回），用于确认报文可解析、可接收。
- 规则：在 `body.ack_msg_id` 中携带被确认消息 ID。

## 5. 心跳与超时机制

1. 心跳方向：客户端主动发送 `ping`，服务端响应 `pong`。
2. 心跳周期：客户端每 1 秒发送一次 `ping`。
3. 客户端超时阈值：连续 5 秒未收到 `pong`，判定连接异常。
4. 服务端超时阈值：连续 5 秒未收到客户端 `ping`，判定连接异常。
5. 异常处理：连接异常后立即触发重连。
6. 重连退避：1s、2s、4s、5s（上限 5s）。

## 6. 通讯层状态码（code）

`code` 仅表达通讯层状态，不得表达业务逻辑结果。

| code | 含义 | 说明 |
|---|---|---|
| 0 | OK | 已接收 |
| 1001 | BAD_FORMAT | JSON 格式错误或字段缺失 |
| 1002 | BAD_TYPE | `type` 非法 |
| 1003 | BAD_API | `api` 非法或未注册 |
| 1004 | UNAUTHORIZED | `identity` 无效或未授权 |
| 1005 | TOO_LARGE | 报文超限 |
| 1006 | RATE_LIMIT | 触发限流 |
| 1007 | DUPLICATE_MSG | 重复 `msg_id` |
| 1010 | INTERNAL_ERROR | 通讯层内部异常 |

## 7. 标准报文示例

## 7.1 payload（业务请求）

```json
{
  "v": "1.0",
  "type": "payload",
  "api": "ZJAPI_PAD_Safe",
  "msg_id": "4e4e7f34-7d7b-4c57-a9d6-3cfbd1c4f3a1",
  "identity": "PAD-DEVICE-0001",
  "code": 0,
  "body": {
    "safe_distance": 20
  },
  "ext": {},
  "ts": 1775966400000
}
```

```json
    "ack_msg_id": "4e4e7f34-7d7b-4c57-a9d6-3cfbd1c4f3a1"
```

```json
}
```

## 8. 与业务接口的映射规则

1. 业务接口名直接填入 `api`，例如 `ZJAPI_Motion_PATH`。
2. 业务参数整体放入 `body`。
3. 各接口不再定义独立外层结构，统一遵循本规范信封。
4. 业务处理结果放在 `body` 内部字段（如 `biz_status`、`biz_msg`），不使用 `code` 表达业务结果。

## 9. 实施要求

1. `msg_id` 全端统一采用uuid。
2. 服务端建立短周期去重缓存（按 `msg_id`）。
3. `ack` 与 `pong` 处理链路应优先级高于普通业务消息。
4. 联调阶段必须校验字段顺序、字段完整性、心跳超时与重连行为。

## 10. 版本演进

- 当前版本：V1.0
- 后续若新增字段，优先通过 `ext` 扩展并保持向后兼容。
